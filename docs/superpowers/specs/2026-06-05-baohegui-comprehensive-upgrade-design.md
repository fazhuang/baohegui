# 包合规全面升级设计文档

> 日期：2026-06-05
> 状态：设计已确认，待实施

## 一、项目背景与目标

### 当前情况

包合规（baohegui）是一个面向招标代理机构和政府采购部门的招标文件发布前合规自检系统。当前系统已有五层审查流水线架构、40+条合规规则、17个AI风险维度，但存在以下问题：

- **核心流水线不完整**：零Token路由审查、参数倾向性检测、四路风险合并器三个关键模块尚未实现
- **审查准确率有待提升**：漏报和误报影响用户信任度
- **用户体验粗糙**：上传流程、审查进度、报告展示等方面不够专业
- **规则覆盖不全面**：行业类型、平台规则覆盖面窄
- **缺乏持续优化机制**：没有反馈回路，规则和模型无法自我改进

### 升级目标

打造行业标杆级的招标文件合规审查产品，在准确率、速度、体验、覆盖面四个维度全面达到生产级质量标准。

## 二、推进策略

### 策略选择：里程碑式推进 + 混合代码策略

- **里程碑式推进**：4个独立可交付的里程碑，每阶段产出可用的端到端增量
- **混合代码策略**：缺失模块全新设计实现，已有模块渐进式优化，前端做大体验重构

## 三、里程碑规划总览

| 里程碑 | 时间 | 重心 | 核心交付 |
|--------|------|------|---------|
| M1 | 第1-4周 | 核心流水线补全 | 五层审查全链路跑通 |
| M2 | 第5-8周 | 规则+LLM增强 | 规则覆盖翻倍 + 可解释性 |
| M3 | 第6-10周 | 前端体验重构 | 全新用户界面和交互 |
| M4 | 第9-12周 | 平台对接+质量闭环 | 多平台覆盖 + 生产级质量 |

> 注：M2和M3有2周重叠期，可并行开发。

## 四、M1：核心流水线补全（第1-4周）

### 目标

补齐五层审查流水线的三个关键缺失模块，让审查引擎从上传到复核全链路完整运转。

### 整体数据流

```
📄 文档上传 → 解析器(第4层) → 章节结构化 + 解析质量评估
                                    ↓
  ⚡ 第0层：零Token路由 ─→ 🔧 第1层：规则引擎 ─→ 🎯 第2层：参数倾向性 ─→ 🧠 第3层：LLM引擎
                                    ↓
                      📊 四路风险合并器(汇总层)
                                    ↓
                      👤 复核状态机 → 最终结论
```

### 4.1 第0层：零Token路由审查 (`engine/routing.py`)

**输入**：预算金额、采购方式、项目类型（从解析器提取的结构化字段）

**路由逻辑**：
- 🟢 绿灯：小额 + 常见采购方式 → 跳过LLM，仅规则引擎快速过
- 🟡 黄灯：中等金额 / 特殊采购方式 → 规则引擎 + LLM关键维度
- 🔴 红灯：大额 / 敏感采购方式 → 五层全开，深度审查

**输出**：交通灯等级 + LLM任务列表（指定哪些维度需要检查）

**设计要点**：
- 规则配置化：阈值可通过配置文件调整
- 零Token消耗：不调用任何LLM
- 可审计：记录每次路由决策的依据

### 4.2 第2层：参数倾向性检测 (`engine/parameter_bias.py`)

**输入**：解析后的技术参数、资格要求字段

**9种违规模式**（基于558个甘肃省政采投诉案例提炼）：

1. **品牌锁定** — 参数恰好匹配某一品牌的技术规格
2. **厂家授权锁** — 要求提供原厂授权函/售后服务承诺函
3. **组合参数指向性** — 多个参数组合后整体指向唯一供应商（最核心、最难检测的模式）
4. **业绩门槛过高** — 业绩要求远超项目规模
5. **资质证书排他** — 要求罕见/特定机构颁发的证书
6. **地域限制** — 变相要求本地化服务
7. **时间窗口过窄** — 不合理的投标准备时间
8. **联合体限制** — 不合理限制联合体投标
9. **分包限制** — 不合理限制分包比例或范围

**关键设计**：组合参数整体指向性是最高级别的隐性排他检测——需要对多个技术参数做关联分析，判断组合后是否形成对某一供应商的唯一指向。结合模板指纹识别和变量标记技术（已有 `template_fingerprint.py` 和 `variable_marker.py`），通过参数偏离度分析实现对隐性排他条款的精准锁定。

### 4.3 汇总层：四路风险合并器 (`engine/fusion.py`)

**输入**：路由结果 + 规则引擎结果 + 参数倾向性结果 + LLM引擎结果 + 解析质量评估

**合并策略**：
- **confirmed**（确定违规）：规则引擎 `forbidden_pattern` 命中 + LLM确认
- **high_risk**（高风险）：规则命中但LLM未确认，或参数倾向性得分超过阈值
- **needs_review**（待人工确认）：仅LLM发现但规则未命中
- **advisory**（提示关注）：轻微风险或信息性提示

**输出**：
- `final_passed`：整体通过 / 不通过
- `final_risk_level`：低 / 中 / 高 / 严重
- `review_status`：auto_passed / auto_failed / needs_review
- `requires_human_review`：是否需要人工复核
- 解析质量置信度调整因子

### 4.4 复核状态机

```
auto_passed  ──→ 结束
auto_failed  ──→ 等待人工复核
needs_review ──→ reviewed_passed ──→ 结束
             ──→ reviewed_failed ──→ 结束
```

### M1 交付物
- [ ] `engine/routing.py` — 零Token路由审查
- [ ] `engine/parameter_bias.py` — 参数倾向性检测（9种模式）
- [ ] `engine/fusion.py` — 四路风险合并器 + 复核状态机
- [ ] 单元测试（每模块 ≥80% 覆盖率）
- [ ] API端点更新（审查流程集成新模块）

## 五、M2：规则+LLM深度增强（第5-8周）

### 目标

大幅提升审查的广度和深度——更多规则、更聪明的LLM、更好的可解释性。

### 5.1 规则库体系化扩展

从当前40+条扩展到约120条，覆盖全采购类型：

| 采购大类 | 子类 | 目标规则数 |
|---------|------|-----------|
| 🏗️ 工程建设 | 房建、市政、交通、水利 | ~40条 |
| 📦 货物采购 | IT设备、医疗、教育、办公 | ~35条 |
| 🛎️ 服务类 | 咨询、物业、IT服务、保险 | ~25条 |
| 📋 通用程序 | 公告、保证金、公示、合同 | ~20条 |
| **合计** | | **~120条** |

每条规则包含：`rule_id`、适用条件（conditional表达式）、检查逻辑、证据定位、法规依据、严重等级。

### 5.2 LLM引擎三项关键升级

#### (A) 审查可解释性（三段式解释）

对每个风险输出：
1. **风险事实** — 文档中哪段文字触发了风险（直接引用原文）
2. **法规依据** — 违反了哪条法规/政策的具体条款
3. **风险推演** — 如果不修改，可能导致什么具体后果

#### (B) 多模型路由

不同场景用不同模型，兼顾质量与成本：

| 场景 | 推荐模型 | 原因 |
|------|---------|------|
| 排他性检测 | DeepSeek | 性价比高，文本理解强 |
| 法律风险评估 | Qwen | 中文法律语义更好 |
| 评分权重合理性 | DeepSeek | 推理能力适合数值分析 |
| 质疑风险预判 | Qwen | 中文语境更准确 |

同时支持手动切换和自动路由配置（按维度+预算）。

#### (C) 结构化输出增强

- **置信度打分**：每个风险附带 0-1 置信度，低于阈值自动降级为疑似风险
- **风险等级自评**：LLM自行评估严重程度，由合并器交叉验证
- **修复建议**：每个风险附带具体的修改建议文本

### 5.3 规则管理后台

为管理员提供可视化规则管理能力：

- **规则编辑器**：可视化条件表达式构建器、规则即时代码预览、测试用例快速验证
- **版本管理**：规则变更历史、一键回滚、变更对比视图
- **规则效能面板**：每条规则的命中率统计、误报/漏报追踪、规则贡献度排名

### M2 交付物
- [ ] 扩展规则库（~120条，覆盖四大类）
- [ ] 三段式可解释性 Prompt 设计 + 实现

- [ ] 结构化输出 Schema 升级

## 六、M3：前端体验重构（第6-10周，与M2并行）

### 目标

从功能可用到专业体验——重塑上传、审查、报告三大核心交互。

### 6.1 文件上传流程重设计

- **拖拽上传区**：大面积拖拽区域，PDF/Word图标预览
- **实时解析进度**：上传 → 解析 → 结构化，三步进度条
- **智能文件检测**：自动识别是否为招标文件，非招标文件给出提示
- **历史文件快速选择**：最近上传列表，一键重新审查
- **批量上传**：支持最多5个文件同时上传排队审查
- **断点续传**：大文件上传支持断点续传

### 6.2 审查进度实时可视化

五层流水线每步进度实时展示：
- 已完成的层：显示 ✓ 标记 + 耗时 + 命中统计
- 正在执行的层：显示加载动画 + 当前操作描述
- 待执行的层：灰色等待状态
- 整体进度条 + 预估剩余时间

### 6.3 交互式合规报告

- **风险热力图**：文档结构树 + 风险标注，一眼看到哪些章节问题最集中
- **原文对照**：每个风险点击后查看原文片段 + 高亮定位
- **一键操作**：下载PDF/Word报告、复制整改清单、分享链接、导出Excel
- **趋势面板**：修订后再次审查时，对比两次结果，展示修复/新增风险

### 6.4 移动端适配

核心功能支持移动端（响应式）：
- 文件上传（支持拍照上传纸质招标文件）
- 审查结果快速查看
- 风险预警推送通知
- 报告分享与审批

> 不是全功能移动端，聚焦"移动办公"场景的核心操作。

- [ ] 审查进度可视化组件（流水线逐层展示）
- [ ] 交互式报告页面（热力图、原文对照、趋势面板）
- [ ] 移动端响应式适配

## 七、M4：平台对接+质量闭环（第9-12周）

### 目标

扩大平台覆盖面，建立持续改进机制，达到生产级质量标准。

### 7.1 多平台规则对接

**通用规则池 + 平台差异层**架构：
- **通用规则**：来自上位法（《招标投标法》《政府采购法》等），所有平台适用
- **平台差异**：每个平台的额外规则和差异化阈值
- **自动匹配**：选择目标平台后，自动叠加通用规则 + 该平台差异规则

覆盖范围：

| 级别 | 平台 | 规则映射 |
|------|------|---------|
| 国家级 | 中国招标投标公共服务平台 | 公告格式、数据规范 |
| 省级 | 甘肃、广东、江苏、浙江、四川 | 各省审查规则差异 |
| 行业 | 住建、交通、水利、卫健 | 行业特殊规则 |

### 7.2 审查反馈闭环

让系统越用越聪明：

**正向反馈**：
- 用户确认风险 → 增加该规则的置信度
- 用户标记遗漏 → 自动创建待补充规则任务
- 好评 → 记录成功案例

**负向反馈**：
- 用户标记误报 → 降低该规则的置信度
- 累积3次误报 → 规则自动进入待审核状态
- 误报率超标 → 触发管理员告警

### 7.3 性能优化目标

| 指标 | 当前 | M4目标 | 手段 |
|------|------|--------|------|
| 单文件审查时间 | ≤3分钟 | ≤1分钟 | LLM并行调用、缓存、预解析 |
| 规则引擎准确率 | ≥95% | ≥97% | 反馈回路持续调优 |
| LLM召回率 | ≥85% | ≥90% | Prompt迭代 + 多模型 |
| LLM误报率 | ≤15% | ≤10% | 融合器交叉验证 |
| 并发审查数 | 未明确 | ≥10并发 | 异步任务队列 |

### 7.4 测试与质量保障

- **测试集**：20+份真实招标文件作为基准测试集
- **规则覆盖**：每条规则至少1个正例 + 1个反例
- **LLM基准**：LLM审查结果人工标注基准
- **回归测试**：每次规则变更自动运行回归套件
- **安全加固**：文件安全扫描、API限流、数据脱敏、完整审计日志

- [ ] 异步任务队列（Celery/ARQ）
- [ ] 性能优化（LLM并行、缓存层）

## 九、技术决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| 推进策略 | 里程碑式 | 每阶段有可交付价值，风险可控 |
| 代码策略 | 混合 | 缺失模块全新设计，已有模块渐进优化 |
| 规则架构 | 通用规则池 + 差异层 | 避免为每个平台维护独立规则集 |
| LLM策略 | 多模型路由 | 不同维度用最合适的模型，兼顾质量和成本 |
| 前端升级 | 大体验重构 | 当前体验是短板，需要系统性地重新设计 |
| 反馈机制 | 结构化闭环 | 正向+负向反馈驱动规则和Prompt持续优化 |
| 测试策略 | 真实文件基准 | 20+份真实招标文件确保审查质量可衡量 |

## 十、风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| LLM审查结果不稳定 | 用户体验差 | 三重校验（Schema → rule_id → 法规依据）、无证据降级 |
| 规则库扩展质量不达标 | 误报率上升 | 每条规则附带测试用例，回归验证 |
| M2/M3并行开发资源冲突 | 进度延误 | 前后端独立开发，API契约先行 |
| 多平台规则差异大 | 开发量超预期 | 先覆盖5个核心省份，再逐步扩展 |
| 真实测试文件获取困难 | 测试不充分 | 公开招标公告下载 + 合作机构提供 |
